Managing live Workflows using BAMOE Management Console

BAMOE Management Console is a web application for viewing the state of all available Business Services and managing and interacting with process instances:

Figure 1. BAMOE Management Console

You can use BAMOE Management Console to manage and interact with process instances, complete User Tasks and manage scheduled Jobs.

In order to be compatible with BAMOE Management Console your Business Services must include the following BAMOE subsystems:

  • Data-Index Add-on: Enables the BAMOE Management Console to access stored events related to processes and domain data from your Business Services. The Data-Index service requires PostgreSQL persistence.

  • Process Management Add-on: Enables the BAMOE Management Console to interact with the Process data from your Business Services through the add-on REST endpoint /management/processes. If you do not enable this add-on for your Business Service, the BAMOE Management Console provides read-only access to your service data without the ability to modify instances, such as aborting Process instances.

To visualize the process instance execution path at the BAMOE Management Console, add the following add-on:

  • Process SVG add-on: Enables the Process SVG diagram to visualize the process instance execution path. If this add-on is not enabled for your Business Service, the BAMOE Management Console does not display the diagram panel.

BAMOE Management Console consists of the following pages that allow you to interact with your process instances and data:

  • Process Instances: List process and subprocess instances and filter them by Process Definition ID/name, Process Instance ID, business key, or status.

  • Jobs: List available jobs, filter them by Process Instance ID, Task ID, Process Definition ID/name, status, and perform actions for any individual one, such as view details, reschedule, or cancel.

  • Tasks: List User Tasks from Processes, filter them by Process Instance ID, Task ID, Process Definition ID/name, status or name, and submit forms to fulfill tasks.

But before you can use these pages, you need to connect your Business Service to the BAMOE Management Console.

Connecting to a Business Service

You will be connected to a Business Service by clicking the + Connect to a runtime… button and filling the required information on the modal:

  • Alias: The name to give your connected Business Service instance (can be anything that helps you identify it).

  • URL: The Business Service root URL (E.g., http://localhost:8080).

  • Force login prompt (for secured runtimes only): Check this if you are already logged in your Identity Provider but would like to log in again (maybe with a different user).

More settings are available in the Advanced OpenID Connect settings section:

  • Client ID: Overrides the Client ID used for this connection. Defaults to the value of the RUNTIME_TOOLS_MANAGEMENT_CONSOLE_OIDC_CLIENT_CLIENT_ID environment variable.

  • Scope: Overrides the scopes requested to the Identity Provider. Useful from some Identity Providers that, for example, will only grant a Refresh Token if the offline_access scope is included in the token request. Defaults to openid email profile.

  • Audience: This is the audience parameter in the Authorization request. Used to identify the service that the token is intended for. Empty by default.

If your Business Service uses OpenID Connect authentication, you should be redirected to the Identity Provider (IdP) login page or, if you’re already logged in, redirected back to the BAMOE Management Console. If your Business Service is unsecured, it should connect directly.

Once logged in, the management pages will be displayed in the side menu, listing Process Instances, Jobs, and Tasks.

Note
 For information on how to secure your Business Service and expose the Indentity Provider URL to the BAMOE Management Console check Securing Business Service API endpoints

Pre-configured connected runtimes

It is possible set up a pre-configured list of runtimes, defined via an environment variable, available via the env.json file. Selecting a different AuthSession updates the state and closes the drawer. This allow users to connect to pre-defined listed runtimes without having to manually configure each one.

Example of a value for the environment variable:

{
    "BAMOE_MANAGEMENT_CONSOLE_CONNECTED_RUNTIMES": [
        {
            "alias": "My Business Service 1",
            "runtimeUrl": "http://localhost:8081",
            "identityProviderUrl": "http://localhost:8181/realms/bamoe"
        },
        {
            "alias": "My Business Service 2",
            "runtimeUrl": "http://localhost:8082",
            "identityProviderUrl": "http://localhost:8182/auth/realms/myrealm",
            "clientId": "bamoe-management-console",
            "scopes": "openid email profile offline_access",
            "audience": "bamoe-service"
        },
        {
            "alias": "My Business Service 3",
            "runtimeUrl": "http://localhost:8083"
        },
        ...
    ]
}

The only required properties are alias and runtimeUrl. The other properties are optional and can be used to configure the connection with an identity provider.

If no identityProviderUrl is provided, the Management Console checks the quarkus-oidc-proxy endpoint to determine the Identity Provider URL (if the runtime is secured).

If no clientId, scopes or audience are provided, the Management Console uses default values.

If multiple runtimes share the same indentityProviderUrl, clientId, scopes, and audience, the Management Console should reuse the same token for all of them.

On the first load, the Management Console will create AuthSessions for each runtime listed in the BAMOE_MANAGEMENT_CONSOLE_CONNECTED_RUNTIMES environment variable. Each runtime will then start authenticating.

If the runtime requires authentication, the Management Console will redirect to the login page, unless the user is already authenticated with the same Identity Provider (using the same clientId, scopes, and audience), in this case the same token can be used for all runtimes.

Overview

The Overview page displays a summary of all Process Instances, grouped by Process Definition ID. This includes basic information such as the name of the Process Definition, number of running instances, completed instances, and failed instances, represented by a bar graph. At the top there should be an aggregated bar graph for all Process Instances.

Process Instances

The Process Instances page shows the list of process instances available. They are displayed in an expandable list, which shows all the child process instances of the particular process instance when expanded.

Figure 2. Process Instances page

Each row in the list contains the information about the process instance, including details like name, status, endpoint etc. The list also provides many ways to filter out the process instances and perform management operations.

Figure 3. Process Details page

The Process Details page consists of the following panels:

  • Process Diagram panel

  • Details panel

  • Process Variables panel

  • Timeline panel

  • Node Trigger panel

  • Milestones panel

  • Jobs panel

Process Diagram panel

This panel enables you to explore the Process diagram and execution path of the related process instance. The execution path is displayed when the BAMOE Management Console finds the .svg resources using the BAMOE Process SVG add-on configuration included in the BAMOE Management Console.

Figure 4. Process Diagram panel

This panel includes different controls to view the Process diagram, such as Selection, Pan, Zoom in, Zoom out, and Fit to viewer.

Details panel

This panel displays the basic Process instance information and provides links to the related Process runtime UI. If the Process instance is related to other Process instances, then links to the parent and subprocess instances are also available in the Details panel.

Figure 5. Details panel

The Details panel consists of the following fields:

  • Name: The Process Definition ID related to the Process instance.

  • Business key: Optional business key of the Process instance.

  • State: The state of the Process instance. The state of a Process instance can be pending, active, completed, aborted, suspended, or error.

  • Id: The unique identifier of the Process instance.

  • Endpoint: The endpoint of the Process instance. You can click the displayed endpoint to view the Process or application UI that triggered the Process instance.

    Figure 6. Process instance details endpoint

  • Start: Time reference indicating when the Process instance is started.

  • Last Updated: Time reference indicating when the Process instance is updated.

  • End: Time reference indicating when the Process instance is completed.

  • Parent Process: Parent Process instance of the instance (if linked). You can click and navigate to the parent Process instance.

  • Sub Processes: Subprocess of the instance (if linked). You can click and navigate to the subprocess instances.

Process Variables panel

The Process Variables panel displays the Process instance domain data in a tree structure. When you hover the mouse on different elements, the available actions are displayed.

Figure 7. Process Variables panel
Figure 8. Actions in Process Variables panel

The process variables in the Process Variables panel consists of the following actions:

  • Expand or collapse: Enables you to expand or collapse an element.

  • Add item: Enables you to add a new key. When you add a new key, it contains null values, which you can edit if required.

    Figure 9. Add new key

  • Copy: Enables you to copy the selected value in clipboard.

  • Edit: Enables you to edit the selected element value.

  • Remove: Enables you to remove the selected element.

If there are pending changes to be saved, the panel displays Changes are not saved yet message. To save the changes that you made to the process variable, click Save, or click Abort to abort the changes. If you want to revert the changes, click the refresh icon.

Timeline panel

The Timeline panel displays the list of node instances related to the Process instance sorted by the start time. Each node consists of an icon, indicating the state of the node, such as active, completed, or error.

Figure 10. Timeline panel

The actions that are available for an active node include retriggering and canceling a node. Whereas the actions for an error node include retrying and skipping the node. The node actions in the Timeline panel are connected with the Business Service to perform the requested action.

If a node in the Timeline panel is associated with a job, then select the clock icon to view the job details or click the menu for additional job operations, such as Job Reschedule or Job Cancel.

Node Trigger panel

The Node Trigger panel displays the list of all Process Definition nodes in a drop-down. When you select a node, the related information is displayed, such as Node name, Node type, and Node id. Click the Trigger button, to trigger the selected node.

Figure 11. Node Trigger panel

The tasks panel shows a list of User Tasks which are available for a process. Each column contains detailed information about the user task which are - Name, Process, Priority, Status, Started and Last update. The columns are sortable.

Tasks Details

Figure 22. Task details

Clicking on the View Details button opens a side panel with more details regarding the current task, including the parent process, possible assigned users/groups, id, name, and state.

The form is autogenerated based on the User Task defined in the parent Process Definition and filling and submitting it will progress the task to the next phase of the task lifecycle.

Impersonation

User Tasks can be assigned to a specific user or user group, and submitting them is limited to that assigned user. But when using a unsecured Business Service or authenticated as a user with impersonation powers, you can submit the task as the assigned user by impersonating them.

Fill the User and/or Group fields in the Impersonation form and click Apply to start impersonating a specific user or user group.

To stop impersonation, click the Reset button.

Note
 When viewing User Tasks from a secured Business Service with a user without impersonation powers, only tasks assigned to the current user or user group can be submitted.
