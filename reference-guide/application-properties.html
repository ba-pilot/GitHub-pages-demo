application.properties Configuration reference

Configuration for an Business Service project is done within the META-INF/application.properties file in both Quarkus and Spring Boot.

This file will also contain other configuration properties for the application.

There are a number of different properties available in a Business Service project, below is a listing of those properties, a short description, the corresponding Java type and a default, if one exists.

Kogito DevServices function only when a Quarkus application is in dev mode. Enabling DevServices starts an in-memory Data-Index container. When using IBM BAMOE, it is recommended to use the data-index addon instead of the in-memory Data-Index container when using DevServices.

quarkus.kogito.devservices.image-name

quarkus.kogito.devservices.shared

(Quarkus Only) Toggle to enable sharing of Quarkus Dev Services. When shared, Quarkus looks for running containers using label-based service discovery. If a matching container is found, it is used, so a second one is not started. Otherwise, Dev Services for a new Data-Index instance is started. The discovery uses the kogito-dev-service-data-index label. The value is configured using the service-name property. Container sharing is only used in dev mode.

boolean

true

quarkus.kogito.devservices.service-name

The value of the kogito-dev-service-data-index label attached to the started container. This property is used when shared is set to true. In this case, before starting a container, Dev Services looks for a container with the kogito-dev-service-data-index label set to the configured value. If found, it will use this container instead of starting a new one. Otherwise, it starts a new container with the kogito-dev-service-data-index label set to the specified value.

string

kogito-data-index

Toggle to enable Kogito Dev Services. Dev Services are generally enabled by default, unless there is an existing configuration present.

boolean

Groups to which the provided user (specified in the name of the config) belongs.

This is only used when testing/developing services through DevUi. The configured user (specified by the user name) and their groups (specified in this configuration) will be listed in a Task Inbox dropdown so the developer can switch between users during development and testing.

Security

Security for a User Task executed through a REST endpoint uses the user/group query parameters to generate the user identity (username and roles) that is used during task operation. However, there are times when a superuser needs to "impersonate" a different user. For example, an admin completing a blocked task on behalf of another user. These two configuration properties enable this "impersonating" to happen.

Firstly, the kogito.security.auth.enabled proper must be set to true. Next, IBM BAMOE will check the roles, configured by the kogito.security.auth.impersonation.allowed-for-roles, that are allowed to be impersonated. This overrides the previously described user identity with the security context.

Roles are defined within the identity provider used by the current application.

Property Name

Description

Type

Default

kogito.security.auth.enabled

Toggle Enabling the application security context when resolving current User Identity.

boolean

false

kogito.security.auth.impersonation.allowed-for-roles

Comma-separated list of roles that allow identity impersonation when resolving the actual User Identity.

These roles should be configured in the Identity Provider the runtime is configured to use.

string
